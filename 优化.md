# tool_search 优化记录

## 原因

- 现有 `tool_search` 容易被模型误当成“内容搜索工具”，把“雷军”“方伟看十年”这类内容关键词直接传进去。
- 失败返回里原先会列出当前可用的 searchable MCP 名单，模型容易把这类信息误读成“虽然失败了，但我已经拿到了某个 MCP 的线索”。
- searchable MCP 的可见元数据原先不够稳定，server 级说明没有稳定进入模型可见层。
- `when_to_use` 和 `description` 语义重叠，会增加维护和展示复杂度。

## 目标

- 把 `tool_search` 收敛成一个稳定的“搜索 searchable MCP 工具名”的路由工具。
- 明确告诉模型：`tool_search` 是搜 MCP 工具名，不是搜内容关键词。
- 让模型在一开始就能看到 searchable MCP 的核心元数据：
  - server 名称
  - alias
  - server description
- 删除会误导模型的失败返回内容，不再在失败时暴露 searchable MCP 名单。
- 保持机制通用，不针对某个 MCP 做特殊路由优化。

## 关键决策

- `tool_search` 失败返回只保留三段：
  - 失败事实
  - 这是搜 MCP 工具的，不是搜内容关键词的
  - 要求模型重新调用 `tool_search`
- searchable MCP 元数据只保留最小必要集合：
  - `alias`
  - `description`
- 撤销 `when_to_use` 字段支持，因为 `description` 足够表达用途。
- searchable MCP 成功命中后仍然按原机制注入真实工具，不改变注入时机。
- BM25 的基本策略不改，仍然用通用文本匹配，不针对 `rednote` 或其他单个 MCP 做特判。
- `rednote` 的 alias 明确收敛为：
  - `xiaohongshu|小红书`
- `rednote` 的 server description 改成中英双语，由 MCP 自己的 `mcp.config.json` 提供，这样 `tool_search` 展示给模型时自然会带出中英描述。

## 大致实现

### 1. tool_search 描述与返回文本收敛

文件：
- `agent/core/tool_loader.py`

主要调整：

- searchable MCP 加载后，缓存结构收敛为：
  - `tools`
  - `alias`
  - `description`
- 新增统一的 searchable MCP 信息格式化逻辑，用于：
  - `tool_search` 的工具描述
  - `tool_search` 成功返回
- `tool_search` 给模型展示 searchable MCP 时，显式显示：
  - 名称
  - `Alias: ...`
  - `Description/描述: ...`
- `tool_search` 成功命中时，返回：
  - `Matched searchable MCP server: ...`
  - `Alias: ...`
  - `Description/描述: ...`
- `tool_search` 失败时，返回文本收敛为：
  - `tool_search failed: no searchable MCP server matched this query.`
  - `tool_search is for finding MCP tools, not content keywords.`
  - `Please call tool_search again with an MCP tool name, not a content keyword.`
- 失败返回中彻底删除：
  - `Available searchable MCP servers`
  - searchable MCP 名单
  - alias 列表

### 2. searchable MCP 元数据收敛

文件：
- `mcp-servers/registry.json`

主要调整：

- `rednote` 的 searchable 元数据只保留：
  - `category`
  - `alias`
- `alias` 明确写成：
  - `xiaohongshu|小红书`
- 不再为 `when_to_use` 保留逻辑入口。

### 3. server description 来源统一

文件：
- `mcp-servers/rednote/mcp.config.json`

主要调整：

- `rednote` 的 server-level `description` 改为中英双语。
- 这条 description 会被 `tool_search` 当作 searchable MCP 的 server description 直接展示给模型。
- 这样无需在 `tool_search` 层面对某个 MCP 做额外特判。

## 结果

- 模型在初始阶段能看到更完整的 searchable MCP 信息，但这些信息只包含 server 元数据，不会直接伪装成已注入工具。
- 失败返回不再提供“可用 MCP 名单”这种半成功线索，减少模型误判。
- `tool_search` 的语义更清晰：
  - 搜 MCP 工具名
  - 不搜内容关键词
- searchable MCP 元数据结构更简单，后续维护成本更低。
- 整套优化保持通用，没有针对 `rednote`、`playwright` 或其他单个 MCP 定制匹配逻辑。

## skill 路径统一到 resources/skills

### 原因

- 安装版里内置 skill 位于应用资源目录，但模型曾被提示 runtime skill 路径，导致读取 skill 附件时会猜到错误目录。
- 旧实现同时加载 `runtime_root/skills` 和 `resources/skills`，模型无法稳定判断真正的 active skill root。
- 文件工具和权限管理原先只放行 `runtime_root`，即使提示改成 `resources/skills`，模型也无法稳定读写 skill 目录。

### 目标

- skill 加载只保留 `resources/skills`
- 模型提示只暴露一个 active skill directory
- skill 附件统一相对当前 `SKILL.md` 所在目录解析
- 保留 `workroot/userdata` 读写能力，并额外允许读写 `resources/skills`
- 不新增专门安装 API，不做 user data fallback

### 大致实现

- `agent/core/paths.py` 新增 `get_skills_root()`，统一提供 active skill root。
- `agent/core/tool_loader.py` 改为只扫描 `get_skills_root()`，并在 skill 返回值中补充 `Skill manifest file` 与 `Skill directory`。
- builtin 文件工具沙箱从单根扩展成双根：`runtime_root + skills_root`。
- `agent/core/permission_manager.py` 把写入/删除越界判断从单一 `workspace_root` 扩展到 `runtime_root + skills_root`。
- `agent/core/main.py` 和 `agent/server/routes/meta.py` 统一移除 runtime skills 文案，只保留单一路径。
- 如果检测到旧 `runtime_root/skills`，后端只记录 warning，不再加载或展示。

### 后续补充

- `resources/playwright-browsers` 也加入模型可写路径，用于浏览器运行时文件管理。
- 模型可读取整个 `resources` 根目录，但写入/删除仍只允许：
  - `resources/skills`
  - `resources/playwright-browsers`
- system prompt 中补充 `resources` 路径和写权限边界，避免模型把“可读整个 resources”误解成“可改整个 resources”。

## Backend Startup Robustness (2026-03-07)

### Reason
- Packaged Windows x64 release showed machine-specific "Backend startup failed" even when bundled python could import required packages.
- Current startup path treated probe failure as hard failure, which is fragile under endpoint security scans/interception and cold-start latency.
- Packaged runtime must not depend on system Python/Node presence.

### Goal
- In packaged mode, backend startup depends only on resources/python and resources/node.
- Improve tolerance for probe timeout and process-chain interception.
- Preserve diagnosability via deterministic startup log file.

### Implementation Overview
- Force packaged Python candidate selection to bundled runtime only.
- Keep bundled Node as required in packaged mode, with explicit missing-runtime error.
- Increase python probe timeout to 100s (SKILLS_MCP_PYTHON_PROBE_TIMEOUT_MS) and make probe non-fatal for startup attempts.
- Add startup state machine: up to 5 retries (SKILLS_MCP_BACKEND_MAX_RETRIES) with exponential backoff base 1s (SKILLS_MCP_BACKEND_RETRY_BASE_MS).
- Add dual launch channels on Windows: direct spawn first, then cmd-wrapper fallback for interception-prone environments.
- Isolate packaged backend environment: clear high-risk inherited vars (PYTHONHOME/PYTHONPATH/VIRTUAL_ENV/CONDA_*/NODE_OPTIONS etc.) and prepend bundled runtime paths.
- Add main-process startup diagnostics log at Electron logs path: ackend-startup.log.

## Local Backend Endpoint Unification (2026-03-08)

### Reason
- Electron main process health check used `127.0.0.1:8000`, while renderer API requests used `localhost:8000`.
- On macOS, `localhost` may resolve differently from `127.0.0.1`, causing renderer requests to fail even when the backend is already healthy.
- The old setup made diagnosis harder because Python startup logs also printed a different host string from the actual bind address.

### Goal
- Keep local backend endpoint definition in one place on the frontend side.
- Pass the same host/port into the Python backend so bind address, health check, and renderer API base stay consistent.
- Avoid platform-specific special cases; fix the address source generically.

### Implementation Overview
- Added a pure shared frontend module for backend host/port/origin/API base/health URL constants.
- Updated Electron main process and renderer API client to import the same shared backend endpoint constants.
- Injected backend host/port into the Python process via environment variables during startup.
- Updated Python backend startup and logs to read the same host/port values from environment variables.

## macOS Bundled Runtime + Startup Reliability (2026-03-08)

### Reason
- macOS first-launch runtime bootstrap still depended on GitHub Release assets and network availability, which is too fragile for non-technical users.
- Even after fixing `localhost`/`127.0.0.1`, fixed port `8000` and startup/UI coupling could still turn backend failures into misleading setup errors.
- Small-scale distribution for known users values offline first-run reliability over installer size.

### Goal
- Bundle macOS runtime directly into the packaged app, matching the Windows delivery model.
- Remove GitHub Release runtime manifest/download from the startup critical path.
- Make backend startup deterministic: loopback-only, dynamic free port, verified health identity, and explicit startup failure UI.

### Implementation Overview
- Removed mac runtime bootstrap from the packaged startup path and bundled `python/node/playwright-browsers/tools` directly into mac app resources.
- Updated GitHub macOS workflow to prepare runtime on runner, package only DMG artifacts, and stop publishing runtime bundle side-assets.
- Added dynamic backend port allocation in Electron main process and passed host/port/app identity into Python through environment variables.
- Updated `/api/health` to return app identity/version and validated that payload before marking backend ready.
- Added preload/runtime status IPC so renderer waits for backend readiness before calling config APIs.
- Added startup failure page with retry/log actions, and changed config-save errors to distinguish backend-unavailable failures from normal validation errors.

## macOS Runtime Packaging Verification Alignment (2026-03-09)

### Reason
- The macOS workflow could build an app that launched, but still miss `runtime/node/node_modules` or usable bundled tools.
- The existing runtime verifier already encoded stricter expectations than the workflow actually prepared.

### Goal
- Make macOS CI prepare the same runtime shape that the packaged app declares in `extraResources`.
- Fail fast on missing core runtime, while keeping non-core tool gaps as warnings.

### Implementation Overview
- Added a dedicated macOS runtime preparation script for GitHub Actions instead of reusing the Windows-oriented PowerShell flow.
- The macOS workflow now prepares `runtime/python`, `runtime/node`, `runtime/playwright-browsers`, and `runtime/tools` before packaging, then runs `npm run verify:runtimes`.
- Updated runtime verification so macOS tool failures are warnings, but Python/Node/Playwright/runtime JS dependency failures still stop the build.
- Fixed packaged runtime path handling so macOS resolves the real runtime root for `node_modules`, Python library paths, and bundled tool payload directories.

## macOS Writable Data Root Isolation (2026-03-09)

### Reason
- Packaged macOS writable data was still falling back to Electron defaults under `~/Library/...`, which mixes project data with unrelated app data.
- This project needs the same explicit data-root boundary concept as Windows, but with a separate macOS location.

### Goal
- Constrain packaged macOS writable data to `~/PaimonData`.
- Keep the only project-related directories on macOS to the install location and `~/PaimonData`.

### Implementation Overview
- Added a macOS data-root resolver in Electron main process with boundary checks for `SKILLS_MCP_DATA_ROOT`.
- Redirected packaged macOS Electron writable paths (`appData`, `userData`, `sessionData`, `cache`, `logs`, `temp`, `crashDumps`) into `~/PaimonData/electron/...`.
- Updated backend environment setup so macOS also receives isolated `HOME`, `TMPDIR`, and `XDG_*` paths under `~/PaimonData`.
- Updated backend shared path helpers so macOS runtime root defaults to `~/PaimonData/workspace-root` and rejects overrides outside that tree.

## Chat Task Polling + Data Root Validation Tightening (2026-03-09)

### Reason
- Frontend chat flow had already switched to `start -> status` polling, but the backend still only exposed the old synchronous `/chat` route, so every request first hit a missing endpoint and then fell back.
- Windows/mac data root validation rules were not fully aligned between Electron and Python, leaving room for split runtime paths or overrides outside the intended boundary.
- Long-running chat requests needed lightweight lifecycle management so in-memory request state does not grow without cleanup.

### Goal
- Expose a real backend polling contract that matches the existing frontend chat client without removing the old synchronous fallback route yet.
- Keep async request state in memory only, with bounded cleanup, instead of introducing a persistent jobs table.
- Make Windows and macOS data-root/runtime-root validation behave the same on both frontend and backend.

### Implementation Overview
- Added an in-memory chat task manager on FastAPI app state and exposed `POST /api/chat/start` plus `GET /api/chat/status/{request_id}`.
- Reused the same event-driven agent execution helper for both background tasks and the legacy synchronous `/api/chat` route, so permission/error/final-response mapping stays consistent.
- Kept `/api/chat/interrupt` semantics unchanged, but marked running task state so polling clients can observe `interrupted` as a terminal status.
- Tightened Windows path validation to reject UNC, relative, and non-`D:` overrides, and aligned macOS `~` expansion/boundary checks between Electron and Python.
- Added cleanup for expired terminal chat tasks so repeated long-running requests do not accumulate unbounded in-memory status records.

## macOS Packaged Backend Probe Shutdown Hardening (2026-03-10)

### Reason
- The packaged macOS verification step could finish Python, Playwright, and backend health checks successfully, then still hang until the GitHub job hit the 3-hour limit.
- The verifier only sent `SIGTERM` to the spawned backend process and returned immediately, without waiting for the child to actually exit.

### Goal
- Make packaged backend verification deterministic end-to-end: after health passes, the verifier must also confirm the temporary backend process exits cleanly.
- Fail fast with actionable diagnostics if backend shutdown hangs, instead of letting CI time out hours later.

### Implementation Overview
- Added a child-process shutdown helper in the runtime verifier that waits for `exit`/`close`, escalates from `SIGTERM` to `SIGKILL`, and records each shutdown attempt.
- Reused the same shutdown path for both success and failure branches of the packaged backend health probe.
- Added explicit shutdown diagnostics to CI logs: signal sent, timeout state, exit code, final signal, and captured stdout/stderr.
- Switched packaged backend probe stdout/stderr capture from live pipes to temp log files so inherited MCP/browser subprocess chains cannot keep the GitHub Actions Node verifier alive just by holding pipe handles open.

## macOS Playwright Compatibility Guardrails (2026-03-10)

### Reason
- The original macOS DMG failure had two separate causes: bundled Python could not start, and bundled Chromium was also too new for the user's macOS 11.7.10 environment.
- The Playwright MCP package had been pulling a prerelease Playwright browser chain, which is too unstable for "best effort old macOS compatibility".
- App startup must not fail just because packaged Playwright browser assets are missing or incompatible.

### Goal
- Keep Python/backend startup as the only hard startup gate.
- Freeze macOS Playwright packaging to a stable browser line and block prerelease Playwright versions from entering release.
- Preserve CI Playwright launch probes as release quality gates without turning Playwright runtime availability into an app-startup blocker.

### Implementation Overview
- Pinned `mcp-servers/playwright` to explicit stable `playwright` and `playwright-core` versions, and added overrides so the packaged browser line no longer follows prerelease transitive dependencies by default.
- Added Playwright version auditing to runtime verification and made prerelease `playwright` / `playwright-core` an immediate mac release failure.
- Extended macOS build and smoke summaries to record `@playwright/mcp`, `playwright`, and `playwright-core` versions plus prerelease audit status.
- Relaxed packaged mac startup so missing bundled Playwright browser assets log a warning and only affect Playwright-dependent features instead of failing backend startup.

## macOS DMG Smoke Mount Path Fix (2026-03-10)

### Reason
- The new dmg smoke-test job was failing before runtime verification even started.
- `hdiutil attach` mounted the dmg under volume names like `/Volumes/Paimon 1.0.6`, but the workflow parsed the mount point with `awk '{print $NF}'`, which drops everything before the final space-separated token.

### Goal
- Make dmg smoke testing robust for normal macOS volume names that contain spaces.
- Keep the smoke test focused on installer validity instead of failing on a shell parsing bug.

### Implementation Overview
- Changed the workflow mount-point extraction to capture the full substring starting at `/Volumes/` instead of the last whitespace-delimited field.
- Added a small diagnostic line so future failures show the resolved mount candidate directly in Actions logs.

## macOS Smoke Realpath Boundary Fix (2026-03-10)

### Reason
- After the dmg mount parsing fix, copied-app smoke verification still failed before runtime probing because the app lived under `/tmp/...`, while macOS canonicalized bundled executable paths under `/private/tmp/...`.
- The verifier compared executable realpaths against a non-canonical allowed root, so valid copied apps were rejected as escaping the packaged runtime boundary.

### Goal
- Keep packaged runtime boundary checks strict without rejecting normal macOS temp-directory aliases.
- Make dmg smoke verification behave the same way as packaged-app verification on the original build output path.

### Implementation Overview
- Updated executable inspection to prefer `fs.realpathSync()` for any existing path, not just symbolic links.
- Canonicalized the allowed root before boundary comparison and included both the original and canonical roots in failure diagnostics.

## macOS Release Asset Collection Flattening (2026-03-10)

### Reason
- The release job downloaded build and smoke artifacts successfully, but `softprops/action-gh-release` still failed because one of the mixed root/nested glob patterns did not match the downloaded directory layout.
- Release publication should not depend on how `download-artifact` happens to flatten or retain subdirectories for a given run.

### Goal
- Make release publication deterministic regardless of artifact directory nesting.
- Preserve strict failure behavior if the expected dmg, checksum, or summary files are actually missing.

### Implementation Overview
- Added a release-side collection step that copies all publishable files into a flat `publish-assets` directory after artifact download.
- Switched the GitHub release upload step to publish from that flat directory only, and kept a hard failure if no publishable files are found.

## Playwright MCP Packaged Stability Hardening (2026-03-09)

### Reason
- Local development and existing Windows runtime use were already stable, so the main remaining risk was packaged-app reliability, especially for Playwright MCP on end-user machines.
- File-presence checks alone were not enough to prove that bundled Node plus bundled Playwright browser assets could actually launch Chromium.
- When packaged Playwright failed on user machines, startup logs did not clearly distinguish missing entrypoints, empty browser assets, or subprocess launch failures.

### Goal
- Block unstable installers at build time if bundled Playwright cannot actually launch.
- Improve packaged startup logs so Playwright-specific failures can be diagnosed from one log file.
- Add a lightweight packaged macOS self-check without turning Playwright issues into full-app startup blockers.

### Implementation Overview
- Extended bundled runtime verification with a real Playwright launch probe that uses bundled Node and bundled browser assets to launch and close headless Chromium.
- Kept that probe build-time only and fail-fast, so packaging stops before shipping a broken installer.
- Added packaged-mode startup logging for Playwright MCP entrypoint, bundled/runtime browser paths, and asset presence.
- Added a packaged macOS Playwright self-check that logs pass/fail details but does not block backend startup or unrelated capabilities.

## macOS Portable Python Packaging Hardening (2026-03-09)

### Reason
- The macOS dmg build was copying the GitHub Actions machine Python into `runtime/python`, which can preserve machine-bound symlinks or framework references.
- That lets CI source-directory checks pass while the final `.app` still fails on user machines with `spawn .../python/bin/python3 ENOENT`.

### Goal
- Bundle a truly portable macOS Python runtime per architecture instead of copying runner-local Python.
- Fail the macOS release workflow if the final packaged `.app` cannot execute bundled Python or start the backend health endpoint.
- Make startup logs distinguish missing executable, broken symlink, and invalid runtime-root resolution.

### Implementation Overview
- Replaced macOS runtime preparation to download a fixed python-build-standalone archive per architecture, verify SHA256, extract it into `runtime/python`, then install runtime requirements there.
- Extended bundled runtime verification to reject broken or out-of-root Python executable links and to optionally verify the final packaged `.app`.
- Added a packaged macOS app verification flow that checks the final bundled Python executable, runs an import probe, then starts the backend and waits for `/api/health`.
- Tightened Electron bundled Python diagnostics so packaged startup failures surface a specific reason before the generic retry loop.

## macOS Packaged Playwright Release Gate (2026-03-09)

### Reason
- The previous macOS packaged-app verification promoted Python to a release gate, but Playwright was still only strongly validated before packaging and only weakly checked at runtime on user machines.
- That left a gap where CI could pass while the final `.app` still failed to launch bundled Chromium from bundled Node after packaging.

### Goal
- Make final `.app` Node + Chromium launchability a hard CI release gate on macOS.
- Stop running the same Playwright self-check by default on end-user startup, so users do not see avoidable runtime noise when CI has already validated the packaged app.

### Implementation Overview
- Refactored Playwright launch verification into a reusable probe that validates Node, Playwright browser assets, MCP entrypoint, and Chromium launch behavior.
- Reused that probe both for pre-package bundled-runtime verification and for post-package `.app` verification against `Contents/Resources/...`.
- Disabled packaged macOS runtime self-check by default and left it behind an explicit `SKILLS_MCP_ENABLE_PLAYWRIGHT_RUNTIME_SELF_CHECK=1` debug switch.

## macOS Clean-Room dmg Smoke Gate (2026-03-09)

### Reason
- Verifying the built `.app` inside the original build workspace is necessary but not sufficient; dmg mounting and app copying can still expose packaging issues that do not appear in-place.
- Release publication should only happen after both the original packaged app and the downloaded dmg payload behave the same way in isolated environments.

### Goal
- Make release publication depend on a second macOS smoke-test job that verifies the dmg payload after download, mount, and copy.
- Keep the verification environment isolated from runner caches and global runtime configuration so CI success reflects the app bundle itself.

### Implementation Overview
- Split the macOS workflow into preparation, per-arch build, per-arch smoke test, and final release jobs.
- Added packaged-only verification mode so smoke tests can validate copied apps from dmg artifacts without requiring local `runtime/` source assets.
- Added isolated probe sandboxes for Playwright and Python import checks to avoid accidental reuse of runner `HOME`, temp, cache, `NODE_PATH`, or Playwright browser cache state.
- Publish release assets only after both architecture smoke tests pass, and attach machine-generated build/smoke summaries for each dmg.
